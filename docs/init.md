# Команда `seomi-ssh init`

## Краткое описание

`init` — единственная реализованная команда пакета. За один интерактивный прогон она:

1. опрашивает серверы в цикле (роль + параметры подключения, повтор до отказа);
2. для каждого сервера запускает SSH-визард (ключ → копирование → проверка → подсказка);
3. пишет реквизиты в `.claude/.env` (merge, без затирания чужих ключей);
4. рендерит managed-блок с картой серверов и примерами в `AGENTS.md` / `CLAUDE.md`;
5. копирует skill `aif-ssh` в `.claude/skills/` проекта.

Команда **идемпотентна**: повторный запуск переиспользует ключ, обновляет (а не дублирует)
env-записи и managed-блок.

## Использование

```bash
seomi-ssh init [--verbose] [--dry-run]
```

| Флаг | Назначение |
|------|------------|
| `--verbose` | Включает DEBUG-логирование (подробный вывод каждого шага). |
| `--dry-run` | Прогоняет опрос серверов, но **не** трогает диск и **не** вызывает SSH; печатает превью managed-блока и список ключей, которые были бы записаны. |

### Что спрашивает опрос

Для каждого сервера по очереди:

| Вопрос | Пример | Обязательность |
|--------|--------|----------------|
| Роль | `prod` / `dev` / своё имя | да |
| Host (домен или IP) | `prod.example.com` | да |
| SSH-пользователь | `ai-agent` | да |
| SSH-порт | `22` (по умолчанию) | нет |
| Путь к приватному ключу | `~/.ssh/id_ed25519` (по умолчанию) | нет |
| Рабочая директория | `/var/www/app` | нет |

После каждого сервера — вопрос «Добавить ещё один сервер?». Отказ завершает цикл.
Поддерживается любое число серверов: только prod; dev + prod; произвольный набор.

### SSH-визард (на каждый сервер)

Цепочка стратегий с graceful degradation:

1. **Keygen** — если ключа по указанному пути нет, генерируется ed25519 (`ssh-keygen -N ''`,
   пустая passphrase — агенту нужен неинтерактивный доступ). Если ключ есть — переиспользуется.
2. **Copy** — `ssh-copy-id` (один раз спросит пароль). Если бинарника нет на PATH (типично
   для Windows OpenSSH) — fallback: публичный ключ передаётся в `~/.ssh/authorized_keys`
   через `ssh`-pipe (с дедупликацией).
3. **Verify** — `ssh -o BatchMode=yes ... 'echo ok'`. BatchMode отключает запрос пароля,
   поэтому ненулевой код = ключ не принят.
4. **Fallback** — при неудачной проверке печатается ручная подсказка с содержимым `.pub`
   и инструкцией добавить ключ через панель хостинга.

Ошибка одного сервера не прерывает прогон — остальные серверы обрабатываются дальше.

## Конфигурация (`.claude/.env`)

Параметры доступа хранятся в `.claude/.env` (gitignored) плоскими ключами с **префиксом
роли**. Один сервер = одна группа ключей:

| Ключ | Назначение | Пишется |
|------|------------|---------|
| `SSH_<PREFIX>_HOST` | домен или IP | всегда |
| `SSH_<PREFIX>_USER` | SSH-пользователь | всегда |
| `SSH_<PREFIX>_PORT` | порт | если задан |
| `SSH_<PREFIX>_KEY`  | путь к приватному ключу | всегда |
| `SSH_<PREFIX>_ROOT` | рабочая директория на сервере | если задана |
| `SSH_SERVERS` | csv префиксов всех серверов (реестр) | всегда |

`<PREFIX>` — это нормализованная роль в `UPPER_SNAKE_CASE`: `prod` → `PROD`,
`staging-eu` → `STAGING_EU`. При повторе роли префикс делается уникальным суффиксом
(`PROD`, затем `PROD_2`). Реестр `SSH_SERVERS` позволяет позже перебрать все серверы.

Пример итогового файла:

```ini
SSH_SERVERS=PROD,DEV

SSH_PROD_HOST=prod.example.com
SSH_PROD_USER=ai-agent
SSH_PROD_PORT=22
SSH_PROD_KEY=~/.ssh/id_ed25519
SSH_PROD_ROOT=/var/www/app

SSH_DEV_HOST=dev.example.com
SSH_DEV_USER=deploy
SSH_DEV_KEY=~/.ssh/id_ed25519
```

> `.claude/.env` — **первое место**, куда агент смотрит за реквизитами доступа: не
> `~/.ssh/config` и не алиасы из других проектов. Пример — в
> `templates/claude-dotenv/.env.example`.

## Изменения в CLI

Эта версия добавляет:

- `seomi-ssh init` — полностью реализована (включая `--dry-run`, `--verbose`);
- `seomi-ssh --help`, `seomi-ssh --version` — справка и версия;
- `seomi-ssh update`, `seomi-ssh doctor` — **заглушки** (выводят предупреждение и выходят
  с кодом 1), зарезервированы для следующих релизов.

## Примеры

```bash
# Обычная настройка
seomi-ssh init

# Прогон без побочных эффектов (превью блока, без SSH и записи на диск)
seomi-ssh init --dry-run

# С подробным логом каждого шага
seomi-ssh init --verbose
```

После настройки агент использует доступ через готовые команды из managed-блока, например:

```bash
ssh -i ~/.ssh/id_ed25519 ai-agent@prod.example.com "<command>"
scp -i ~/.ssh/id_ed25519 -r ./local-path ai-agent@prod.example.com:/var/www/app/
rsync -avz -e "ssh -i ~/.ssh/id_ed25519" ./local-dir/ ai-agent@prod.example.com:/var/www/app/
```

## Диагностика

| Симптом | Причина / решение |
|---------|-------------------|
| `verified=false`, печатается ручная подсказка | Хостинг не принимает ключ автоматически (часто shared-хостинги). Добавьте содержимое `.pub` вручную через панель или в `~/.ssh/authorized_keys`, затем повторите `init`. |
| `ssh-copy-id missing — using ssh fallback` | На Windows OpenSSH нет `ssh-copy-id` — это нормально, сработает ssh-pipe fallback. |
| `ssh-keygen failed` | `ssh-keygen` не найден в PATH. Установите OpenSSH client. |
| Блок не записан в `AGENTS.md`/`CLAUDE.md` | На шаге выбора файла выбран «skip». Запустите `init` снова и выберите целевой файл. |
| Реквизиты другого сервера пропали | Не должно происходить: `.claude/.env` мержится, чужие ключи сохраняются. Проверьте, что файл не редактировался вручную с дублями. |
| Путь с `~` не разворачивается | `init` разворачивает `~` сам; если задавали путь вручную в `.env`, используйте абсолютный путь или `~/...`. |
